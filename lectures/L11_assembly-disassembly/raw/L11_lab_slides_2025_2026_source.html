Welcome to Lecture 11 Lab: The Disassembly View

Peeking Under the Hood

  • This lab is your opportunity to see how your high-level C code translates into low-level machine instructions.
  • We will use the debugger’s disassembly window to observe the CPU’s actual operations.

Learning Objectives for Lab 10

  • Navigate and interpret the debugger’s disassembly view.
  • Map C source code lines to corresponding ARM assembly instructions.
  • Observe the execution flow at the assembly level.
  • Understand how C constructs (variables, loops, functions) are implemented in assembly.
  • Utilize the disassembly view for advanced debugging and optimization insights.

Conceptual Background: Why Disassembly?

The Deeper Understanding

  • Debugging Complex Issues: When C code behaves unexpectedly, assembly can reveal the true execution path.
  • Performance Optimization: Identify bottlenecks and inefficient code sequences.
  • Understanding Compiler Behavior: See how the compiler translates your C code, including optimizations.
  • Hardware Interaction: Gain a clearer picture of how the CPU directly manipulates registers and memory.

Recap: C to Assembly Mapping

The Compiler’s Job

  • The C compiler translates your human-readable C code into machine-executable assembly instructions.
  • Each C statement can translate into multiple assembly instructions.
  • Variables become memory locations or registers.
  • Functions become sequences of instructions with specific calling conventions.

Step-by-Step Procedure: Set a Breakpoint

Pausing Execution

  • Open your main.c file.
  • Choose a line of C code where you want to pause execution.
    • Good candidates: inside a loop, before/after a function call, or an arithmetic operation.
  • Double-click on the left margin next to the line number to set a breakpoint (a blue dot will appear).

Step-by-Step Procedure: Open Disassembly View

Revealing the Machine Code

  • Once in the Debug perspective, go to Window -> Show View -> Disassembly.
  • The Disassembly view will open, showing the assembly instructions corresponding to your C code.
  • You will see the C source code interleaved with the assembly instructions.

Visual Suggestion

Interpreting the Disassembly View

Columns Explained

  • Address: Memory address of the instruction.
  • Machine Code: The raw hexadecimal representation of the instruction.
  • Assembly Instruction: The mnemonic for the instruction (e.g., MOV, LDR, ADD).
  • Operands: The registers, memory addresses, or immediate values the instruction operates on.
  • Source Code: The corresponding line of C code (if available).

Stepping Through Code: C vs. Assembly

Observing Execution Flow

  • Use the debugger’s step controls:
    • Step Over (F6): Executes the current C line (or assembly instruction) and moves to the next.
    • Step Into (F5): Enters a function call (both C and assembly).
    • Step Out (F7): Executes until the current function returns.

Exercise

  • Step through a few lines of your C code.
  • Observe how the program counter (PC) moves in the Disassembly view.
  • Notice how a single C line might correspond to multiple assembly instructions.

Observing Register Changes

The CPU’s Internal State

  • Open the Registers view (Window -> Show View -> Registers).
  • As you step through assembly instructions, observe how the values in the general-purpose registers (R0-R12) and special registers (SP, LR, PC) change.
  • This is crucial for understanding data flow within the CPU.

Data in RAM

Analyzing C Constructs: Variable Assignment

C Code

int a = 10;
int b = 20;
int sum = a + b;

Disassembly Focus

  • Look for MOV instructions (Move) to load immediate values into registers.
  • Observe STR (Store) instructions to save register values to memory (stack).
  • Then LDR (Load) to retrieve values from memory into registers for operations.

Analyzing C Constructs: Arithmetic Operations

C Code

result = val1 * val2 + offset;

Disassembly Focus

  • Identify MUL (Multiply) and ADD (Add) instructions.
  • Notice how intermediate results are held in registers.
  • Observe the order of operations as translated by the compiler.

Analyzing C Constructs: if/else Statements

C Code

if (temperature > 25) {
    status = HOT;
} else {
    status = COLD;
}

Disassembly Focus

  • Look for CMP (Compare) instructions, which set condition flags.
  • Identify conditional branch instructions like BGT (Branch if Greater Than), BLE (Branch if Less than or Equal).
  • Observe the jump to the else block or skipping it.

Analyzing C Constructs: for Loops

C Code

for (int i = 0; i < 10; i++) {
    // loop body
}

Disassembly Focus

  • Identify the loop initialization (e.g., MOV R0, #0).
  • Locate the loop condition check (CMP, BGE or BLT).
  • Observe the increment/decrement (ADD, SUB).
  • Find the unconditional branch (B) that jumps back to the loop start.

Analyzing C Constructs: Function Calls

C Code

void setup() {
    init_gpio();
    // ...
}

Disassembly Focus

  • When init_gpio() is called, observe the BL (Branch with Link) instruction.
  • Notice how the Link Register (LR) is updated with the return address.
  • Inside init_gpio(), look for BX LR (Branch and Exchange to LR) to return to the caller.

Analyzing C Constructs: Pointers

C Code

int value = 100;
int *ptr = &value;
*ptr = 200;

Disassembly Focus

  • Observe LDR instructions for loading the address of value into ptr.
  • When *ptr = 200; is executed, look for STR instructions that use the address in ptr to write 200 to memory.

Common ARM Assembly Instructions (Review)

Data Movement

  • MOV: Move data between registers or immediate to register.
  • LDR: Load data from memory to register.
  • STR: Store data from register to memory.

Arithmetic

  • ADD: Addition.
  • SUB: Subtraction.
  • MUL: Multiplication.

Common ARM Assembly Instructions (Review Cont.)

Logical & Bitwise

  • AND, ORR, EOR: Bitwise logical operations.
  • LSL, LSR, ASR: Shift operations.

Control Flow

  • CMP: Compare values, sets flags.
  • B: Unconditional branch.
  • BL: Branch with Link (for function calls).
  • BEQ, BNE, BGT, BLT: Conditional branches.

Debugging Tips with Disassembly

Advanced Troubleshooting

  • Compiler Optimizations: When optimizations are enabled, C code lines might not map directly to assembly. The debugger might jump around.
    • Temporarily disable optimizations for easier debugging if needed.
  • Stack Analysis: Use the Registers view (SP) and Memory view to understand stack usage.
  • Interrupt Context: When an interrupt occurs, observe how registers are pushed onto the stack.

Identifying Issues with Disassembly

Real-World Scenarios

  • Infinite Loops: Identify the unconditional branch that keeps the PC stuck in a loop.
  • Incorrect Pointer Usage: See if LDR/STR instructions are accessing unexpected memory addresses.
  • Function Call Errors: Verify that BL and BX LR are used correctly for function entry/exit.
  • Unexpected Register Values: Track how data flows through registers.
