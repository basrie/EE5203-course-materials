The code you never wrote

You wrote counter++;. The chip has never heard of it. This is what it actually runs—and your debugger will show it to you.

080004a4:  4b04     ldr   r3, [pc, #16]    ; address of counter
080004a6:  681b     ldr   r3, [r3, #0]     ; fetch its value
080004a8:  3301     adds  r3, #1           ; add one
080004aa:  4a03     ldr   r2, [pc, #12]    ; the address again
080004ac:  6013     str   r3, [r2, #0]     ; write it back

By the end, you will read listings like this fluently—and use them to see exactly what the compiler did to every line of C you have written this term.

Every listing in this deck is real output from arm-none-eabi-gcc 14.3 — the exact toolchain inside STM32CubeCLT that VS Code drives. Nothing is invented.

Load/store architecture

01 Only two instruction families touch memory: LDR (memory → register) and STR (register → memory).

02 Everything else—math, logic, compares—works register to register.

03 So every C statement becomes the same rhythm: load, compute, store.

You saw the rhythm on the opener: load counter, add one, store counter. Once you hear it, every listing gets easier.

Deep dive: PM0214 Rev 10, §3.4 Memory access instructions — p. 69.

Anatomy of an instruction

Read any instruction the same way: operation, destination, operands — and an s means the flags learn about the result.

MOV and constants

movs r3, #1              ; r3 = 1  (small constants fit inside)
movs r0, #20             ; argument setup, as we will see
ldr  r3, [pc, #16]       ; big constants live in a literal pool
                         ;   -> loads 0x40020014 (GPIOA ODR!)

01 Small numbers travel inside the instruction—an immediate.

02 A 32-bit address cannot fit in a 16-bit instruction—the compiler parks it after the function and loads it PC-relative.

Deep dive: PM0214 Rev 10, §3.5.6 MOV and MVN — p. 89.

LDR and STR

ldr r3, [r7, #4]     ; r3 = memory[r7 + 4]     (a local variable!)
str r0, [r7, #4]     ; memory[r7 + 4] = r0
ldr r3, [r3, #0]     ; r3 = memory[r3]         (follow a pointer)
str r3, [r2, #0]     ; memory[r2] = r3

01 Brackets mean memory at this address—r3 and [r3] are entirely different things.

02 At -O0, [r7, #4] is your local variable: r7 frames the stack, #4 is its slot.

03 L03’s pointers, made visible: dereferencing is exactly one ldr.

Deep dive: PM0214 Rev 10, §3.4.2 LDR and STR, immediate offset — p. 71.

counter++, for real

The compiler’s actual output (-O0), prologue trimmed:

ldr  r3, [pc, #16]     ; r3 = &counter   (from the literal pool)
ldr  r3, [r3, #0]      ; r3 = counter
adds r3, #1            ; r3 = r3 + 1
ldr  r2, [pc, #12]     ; r2 = &counter   (again)
str  r3, [r2, #0]      ; counter = r3

LOAD, COMPUTE, STORE. Four instructions for one ++—and because counter is volatile, the compiler is forbidden to cache it in a register between uses. L04’s keyword, explained at last from below.

Arithmetic and logic

add  r3, r2            ; r3 = r3 + r2
lsls r3, r3, #1        ; r3 = r3 << 1     (the compiler's *2)
orr.w r3, r3, #32      ; r3 = r3 | 32     -- there it is: |= (1u << 5)
bic  r3, r3, #32       ; r3 = r3 & ~32    -- and &= ~(1u << 5)

01 The L04 masking idioms are single instructions: ORR sets, BIC clears, EOR toggles.

02 Multiplication by powers of two compiles to shifts—2*x became lsls.

03 orr.w is a 32-bit Thumb-2 encoding—the .w just means wide.

Deep dive: PM0214 Rev 10, §3.5.1 ADD, SUB… — p. 83, §3.5.2 AND, ORR, EOR, BIC… — p. 85, §3.3.4 Shift operations — p. 62.

Practice: run three instructions

Registers before: r3 = 0.

1 movs r3, #5 — now r3 = ?

2 lsls r3, r3, #2 — now r3 = ?

3 adds r3, #1 — now r3 = ? And which C expression did you just run?

5 → 20 → 21. You just executed 4*x + 1 the way the chip does—and all three, ending in s, updated the flags.

CMP and the flags

cmp  r3, #10       ; compute r3 - 10, keep only the flags
                   ;   result thrown away, N Z C V remembered

01 CMP is a subtraction that publishes only its verdict: the four flags.

02 Branches then read the verdict: beq/bne for equality, bgt/blt for signed, bhi/bls for unsigned.

03 Our uint32_t value > 10 compiled to bls—the compiler chose the unsigned family because you chose the type.

Deep dive: PM0214 Rev 10, §3.5.5 CMP and CMN — p. 88, §3.3.7 Conditional execution — p. 65.

The if, for real

if (value > 10) { result = 1; } else { result = 0; }

The actual -O0 output (value in [r7,#4], result in [r7,#12]):

ldr  r3, [r7, #4]      ; load value
cmp  r3, #10           ; compare with 10
bls  else_part         ; if value <= 10 (unsigned), jump
movs r3, #1            ; then-branch: result = 1
str  r3, [r7, #12]
b    after_if          ; skip the else
else_part:  movs r3, #0
            str  r3, [r7, #12]

Every if is this shape: compare, branch around the code you skip. The condition is inverted—the branch jumps when the if is false.

The loop, for real

for (uint32_t i = 1; i <= n; i++) { total += i; }
      b    check          ; jump straight to the test
body: ldr  r2, [r7, #12]  ; total
      ldr  r3, [r7, #8]   ; i
      add  r3, r2
      str  r3, [r7, #12]  ; total += i   (then i++, same rhythm)
check: ldr  r2, [r7, #8]   ; i
      ldr  r3, [r7, #4]   ; n
      cmp  r2, r3         ; i <= n ?
      bls  body           ; yes: go around again

A loop is just a backward branch. Spot a branch to a lower address in any listing and you have found the loop.

BL and the way home

uint32_t use_scale(void) { return scale(20); }
push {r7, lr}          ; save the old way home
movs r0, #20           ; argument goes in r0
bl   scale             ; call: LR = return address, PC = scale
mov  r3, r0            ; result comes back in r0
pop  {r7, pc}          ; popping into PC *is* the return

01 The calling convention: arguments in r0–r3, result in r0. Now L03’s “return value” has an address.

02 pop {…, pc} restores the saved lr straight into pc—return and cleanup in one instruction.

Deep dive: PM0214 Rev 10, B, BL, BX — p. 142, §3.4.7 PUSH and POP — p. 78.

Practice: trace the call

Step through use_scale() above, one instruction at a time:

1 Just after bl scale — what is in r0, and what is in lr?

2 scale runs lsls r0, r0, #1 then adds r0, #1 — what comes back?

3 What exactly does pop {r7, pc} do?

r0 = 20, lr = the address after the bl; the result 41 returns in r0; and the pop restores r7, then loads the saved return address into pc—execution is home.

What the optimizer did

The same use_scale() at -O2—this is the entire function:

movs r0, #41           ; 2*20 + 1, computed at compile time
bx   lr

01 The call, the argument, the multiply—gone. The compiler proved the answer is 41.

02 Our if became ite ls / movls / movhi—conditional execution, no branch at all.

03 This is why debugging -O2 feels haunted: lines vanish, steps jump. Debug builds use -O0 on purpose.

volatile is the exception the optimizer must respect—your counter++ stayed a real load-add-store even at -O2. Peripheral registers survive optimization because of that keyword.

The Disassembly view

01 In a debug session: right-click the source → Open Disassembly View—your C interleaved with its instructions.

02 The i→ toolbar button—Instruction Stepping Mode—makes Step Into advance one instruction, not one line.

03 The Registers view shows r0–r12, sp, lr, pc, xpsr—live, changed values highlighted.

04 Works on any code—yours, HAL’s, code compiled without source.

Register evidence

01 Step one adds r3, #1: r3 increments and the flags in xpsr react.

02 Watch pc advance by 2—then by 4 across a .w instruction. Thumb-2, live.

03 Step a push {r7, lr}: sp drops by 8; find your return address in the Memory view at the new sp.

04 Step the bl: pc teleports, lr catches the way home.

A successful build proves none of this. The Registers view does.

Common misreadings

01 BRACKETS IGNORED: r3 is a value; [r3] is memory at that address.

02 FLAGS ASSUMED: only cmp/tst and s-suffixes write flags—add no, adds yes.

03 WRONG SIGN FAMILY: bgt after an unsigned compare—silently wrong for half the values.

04 BRANCH LOGIC INVERTED: the compiler branches when your if is false.

05 -O2 PANIC: vanishing lines and jumping steps are the optimizer, not a broken debugger.

06 ONE LINE ≠ ONE INSTRUCTION: counter++ was four; scale(20) was zero.
