Toolchain reasoning, prediction, and debugging

Author

Basri Erdogan

Expected time: 30-45 minutes
Policy: Attempt every question before consulting the worked solutions.

Part A - Core concepts

  1. For each item, classify it as a processor, microcontroller, peripheral, development board, or embedded system. Some descriptions may contain more than one relevant level; state the most useful classification.

    1. Arm Cortex-M4
    2. STM32F401RE
    3. Nucleo-F401RE
    4. TIM2
    5. A thermostat containing an STM32 board, temperature sensor, and relay

  2. Give one example of each constraint:

    1. timing constraint;
    2. memory constraint;
    3. energy constraint;
    4. safety or reliability constraint.

  3. A project builds with zero errors, but the LED never changes. Explain why this is not a contradiction.

Part B - Toolchain sequencing

  1. Put these actions in a valid order:

    • flash firmware;
    • link object files;
    • edit main.c;
    • compile source files;
    • observe the LED;
    • inspect a variable at a breakpoint.

  2. Match each symptom to the most likely initial investigation.

Symptom Investigation choices
expected ';' before '}' A. Debug connection/USB
undefined reference to process_sample B. Compile-time syntax
No ST-LINK detected C. Link-time definition
Program pauses but variable is wrong D. Run-time logic/state

  1. What evidence would you record for each claim?

    1. The source compiled.
    2. The firmware was written to the board.
    3. The loop executed repeatedly.
    4. The physical output met the timing requirement.

Part C - Predict and trace

  1. Trace the code without running it.
int count = 0;

while (count < 3) {
    count = count + 1;
}
  1. List count after each loop body execution.
  2. What is its final value?
  3. On which line would you place a breakpoint to observe the value changing?
  1. Predict the observable difference between the two loops.
/* Program A */                   /* Program B */
while (1)                         while (1)
{                                 {
    HAL_GPIO_TogglePin(...);          HAL_GPIO_TogglePin(...);
    HAL_Delay(500);                   HAL_Delay(100);
}                                 }

What remains identical? What changes? What measurement would support your claim?

  1. A student changes HAL_Delay(500) to HAL_Delay(100), presses Run, and observes no change. List four distinct explanations and give one test for each.

Part D - Debugging decisions

  1. Rewrite “My code does not work” as a useful debugging report using:

    • expected behavior;
    • observed behavior;
    • evidence;
    • next test.

  2. A breakpoint placed on HAL_GPIO_TogglePin(...) is never reached. Rank the following checks in a sensible order and justify the first two:

    • confirm the correct project was flashed;
    • replace the USB cable;
    • confirm the program reached main;
    • inspect whether execution is paused elsewhere;
    • check the loop and branch conditions.

  3. The LED blinks correctly, but the debugger displays an unexpected local variable value. Give two reasons the LED result alone cannot prove the variable is correct.
